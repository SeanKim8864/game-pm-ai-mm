성과평가 방향과 측정 기준

AI를 몇 번 썼는지를 세는 방식은 금방 한계가 온다. 평가는 결국 업무가 얼마나 빨라졌는지, 판단이 얼마나 좋아졌는지, 그리고 그 변화가 실제 사업 지표나 의사결정에 어떤 도움을 줬는지로 봐야 한다.

1방향성: 속도와 판단력

게임사업PM 업무에서 AI 활용 성과는 크게 두 갈래로 나눠 보는 게 현실적이다. 반복 업무를 줄이는 것, 그리고 같은 시간 안에 더 나은 판단을 하는 것.

Efficiency

같은 일을 더 빠르게

리포트, 공지, 회의록, 경쟁작 조사처럼 반복되는 업무의 시간을 줄인다. 다만 “빨라졌다”만으로 끝내면 약하다.

  • 주간 매출/DAU 리포트: 4시간 → 30분
  • 패치노트 기반 공지 초안: 2시간 → 20분
  • 경쟁작 순위·리뷰·뉴스 요약: 3시간 → 30분
  • 이벤트 전후 성과 정리: 3시간 → 40분
  • 회의록 정리와 액션아이템 추출: 1시간 → 10분
평가 포인트
절약된 시간을 유저 세그먼트 분석, BM 검토, 개발팀 협업처럼 더 중요한 일에 다시 썼는지 확인한다.
Quality

같은 시간에 더 깊게 보기

단순 요약을 넘어 결제 패턴, 코호트, VOC, 소재 성과를 함께 보며 의사결정의 근거를 두껍게 만든다.

  • 매출 추이 확인 → 결제 패턴·ARPU 이상 징후까지 확인
  • 경험 기반 가격 책정 → 가격대별 매출·전환율 시나리오 비교
  • 유사 업데이트 1~2건 비교 → 과거 업데이트 전체 패턴 확인
  • CS 주요 이슈 확인 → 커뮤니티·앱리뷰·CS 감성 흐름 통합
  • 소재 감각 검토 → CTR/CPI 데이터 기반 피드백
평가 포인트
AI가 낸 결과를 그대로 붙인 것이 아니라, PM이 해석해서 실행 가능한 시책으로 바꿨는지를 본다.

3평가 등급 기준

등급은 도구 사용 빈도가 아니라 업무 변화의 크기로 나눈다. 특히 S/A는 증빙 가능한 사례가 있어야 한다.

S

비즈니스를 바꿨다

AI 분석이 매출·리텐션·전환율 같은 지표 개선에 직접 기여했고, 그 방식이 팀 표준으로 남았다.

  • 측정 가능한 KPI 개선
  • 조직 차원의 가이드·교육 주도
  • 반복 가능한 프로세스화
A

업무의 일부가 됐다

정기 업무 여러 개에 안정적으로 붙었고, 실제 의사결정에도 반복적으로 쓰인다.

  • 정기 업무 3건 이상 적용
  • 50% 이상 절감 업무 2건 이상
  • 분기 의사결정 2건 이상 반영
B

쓰기 시작했다

일부 업무에서 편해진 것은 확인됐지만 아직 개인의 습관 수준이고, 팀 프로세스까지는 가지 못했다.

  • 간헐적 활용
  • 효율 개선은 있으나 비정기적
  • 성과 기록이 부족함
C

성과로 이어지지 않았다

사용 이력은 있으나 업무 성과와 연결되지 않았거나, 검증 없이 사용해 품질 문제가 생겼다.

  • 업무 성과 증빙 없음
  • 부정확한 산출물 사용
  • 검증 체계 부재

4MM 반영 시 원칙

평가 항목으로 넣을수록 더 조심해야 한다. “많이 썼다”가 높은 점수가 되면 금방 형식적인 보고가 된다.

사용 여부가 아니라 활용 성과를 본다

주 5회 사용 같은 빈도는 참고값일 뿐이다. 경쟁작 분석 시간을 70% 줄였고, 그 시간으로 이탈 패턴 분석을 새로 했다면 그게 성과다.

Before/After를 먼저 남긴다

평가가 시작된 뒤에 기억으로 적으면 흔들린다. 업무별 현재 소요시간, 산출물 수준, 활용 계획을 초기에 기록해둔다.

절감 시간의 재투자까지 묶어서 본다

예를 들어 주 7시간을 줄였다면, 그 시간이 LTV 분석, BM 시뮬레이션, 개발팀 협업에 어떻게 쓰였는지 같이 적는다.

절감: 리포트 3h + 공지 1.5h + 경쟁작 조사 2.5h
재투자: 세그먼트 LTV 3h + 가격 시뮬레이션 2h + 협업 미팅 2h

검증 체크리스트는 필수다

수치가 원본과 맞는지, 빠진 맥락은 없는지, 도메인 지식과 충돌하지 않는지 확인한다. 오류가 나오면 프롬프트와 데이터 입력 방식을 같이 고친다.

개인 성과와 조직 기여를 나눠 본다

개인 성과 70%, 조직 기여 30% 정도로 나누는 안이 현실적이다. 팀 가이드, 프롬프트 템플릿, 교육, 실패 사례 공유도 기여로 본다.

문서에는 ‘사례’가 남아야 한다

좋은 평가는 문장보다 증빙이 강하다. 어떤 업무에 썼고, 무엇을 바꿨고, 결과가 어땠는지를 한 사례 단위로 남긴다.

5실행 로드맵

처음부터 전 업무에 적용하려고 하면 흐지부지된다. 리포트와 경쟁작 분석처럼 반복성이 높은 업무부터 시작하는 편이 낫다.
